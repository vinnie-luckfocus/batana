# PRD: UI 重新设计

## Executive Summary

batana 棒球挥棒分析系统当前 UI 存在视觉过时、控件重叠、交互不友好等问题，影响用户体验。本次重新设计旨在全面提升产品的视觉呈现和交互体验，面向业余棒球爱好者，打造一个兼具科技感与艺术感的专业友好界面。

设计将采用 Flutter-Neumorphic 风格，保留现有核心功能的同时，新增从相册选择视频进行分析的能力，覆盖所有页面和模块，为用户提供流畅、直观、现代化的使用体验。

## Problem Statement

### 当前问题
1. **视觉过时**：界面设计陈旧，缺乏现代感，无法吸引和留住用户
2. **控件重叠**：布局混乱，控件之间存在重叠，影响可读性和可操作性
3. **交互不友好**：操作流程不够直观，学习成本高，用户容易迷失

### 为什么现在重要
- MVP 阶段已验证核心功能价值，但用户留存率受限于体验问题
- 业余爱好者对界面友好度要求高，专业感与易用性需要平衡
- 竞品不断优化体验，需要通过设计差异化建立竞争优势

## User Stories

### 主要用户画像
**业余棒球爱好者 - 小李**
- 年龄：25-35 岁
- 技术水平：中等，熟悉移动应用
- 使用场景：周末训练后分析挥棒动作，寻求改进建议
- 痛点：现有界面不够直观，找不到想要的功能，视觉体验差

### 用户旅程

#### 旅程 1：首次使用
**作为** 新用户
**我想要** 快速理解应用功能并完成首次分析
**以便于** 评估产品价值并决定是否继续使用

**用户流程**：
1. 打开应用 → 看到清晰的欢迎引导
2. 点击"开始分析" → 选择录制新视频或从相册选择
3. 完成录制/选择 → 自动开始分析
4. 查看结果 → 直观的评分和建议展示

**接受标准**：
- 首次使用无需查看帮助文档即可完成分析
- 关键操作按钮清晰可见，视觉层级明确
- 引导流程不超过 3 步

#### 旅程 2：日常训练分析
**作为** 老用户
**我想要** 快速录制并分析挥棒动作
**以便于** 及时获得反馈并调整动作

**用户流程**：
1. 打开应用 → 直接进入主界面
2. 点击录制按钮 → 开始录制
3. 完成录制 → 查看实时分析进度
4. 查看结果 → 对比历史记录

**接受标准**：
- 从打开应用到开始录制不超过 2 次点击
- 分析进度可视化，用户清楚当前状态
- 历史记录易于访问和对比

#### 旅程 3：从相册分析视频
**作为** 用户
**我想要** 分析之前录制的训练视频
**以便于** 回顾和分析过往表现

**用户流程**：
1. 打开应用 → 点击"从相册选择"
2. 浏览相册 → 选择目标视频
3. 确认选择 → 开始分析
4. 查看结果 → 保存到历史记录

**接受标准**：
- 支持常见视频格式（MP4, MOV 等）
- 视频选择界面清晰，支持预览
- 分析流程与录制视频一致

## Requirements

### Functional Requirements

#### FR1: 主界面重新设计
- 清晰的功能入口：录制视频、选择相册、历史记录
- 快速访问最近分析结果
- 简洁的导航结构

#### FR2: 录制界面优化
- 实时取景预览
- 清晰的录制控制按钮（开始/停止/重录）
- 录制时长提示
- 网格辅助线（可选）

#### FR3: 相册视频选择（新功能）
- 集成系统相册选择器
- 支持视频预览
- 显示视频时长和分辨率
- 支持常见格式：MP4, MOV, AVI

#### FR4: 分析进度界面
- 可视化分析进度条
- 当前处理阶段说明（姿态识别、指标计算、评分生成）
- 预计剩余时间
- 支持取消操作

#### FR5: 结果展示界面
- 综合评分（大号显示，视觉突出）
- 分项评分（挥棒速度、身体协调、击球点等）
- 关键帧展示（可滑动查看）
- 改进建议（分点列出，易于理解）
- 分享和保存功能

#### FR6: 历史记录界面
- 时间线展示历史分析
- 支持筛选和排序
- 快速对比功能
- 删除和导出功能

#### FR7: 设置界面
- 用户偏好设置
- 相机权限管理
- 存储空间管理
- 关于和帮助

### Non-Functional Requirements

#### NFR1: 性能
- 界面响应时间 < 100ms
- 页面切换动画流畅（60fps）
- 视频加载时间 < 2 秒
- 分析完成时间 ≤ 15 秒（MVP 目标）

#### NFR2: 视觉设计
- 采用 Flutter-Neumorphic 设计风格
- 统一的色彩系统（主色、辅助色、强调色）
- 一致的字体层级和间距
- 适配深色模式（可选）

#### NFR3: 交互体验
- 所有交互提供视觉反馈（按钮点击、加载状态）
- 错误提示清晰友好
- 支持手势操作（滑动、缩放）
- 无障碍支持（VoiceOver/TalkBack）

#### NFR4: 兼容性
- 支持 iOS 13+ 和 Android 8+
- 适配不同屏幕尺寸（iPhone SE 到 iPad）
- 横竖屏自适应

#### NFR5: 可维护性
- 组件化设计，便于复用
- 统一的设计系统（Design System）
- 清晰的代码结构和注释

## Success Criteria

### 定量指标
1. **用户留存率**：D7 留存率从 25% 提升至 40%
2. **任务完成率**：首次分析完成率 ≥ 90%
3. **操作效率**：从打开应用到开始分析的平均时间 < 30 秒
4. **用户满意度**：应用商店评分从 3.5 提升至 4.5+
5. **崩溃率**：应用崩溃率 < 0.5%

### 定性指标
1. **视觉现代化**：用户反馈界面"现代"、"专业"
2. **易用性**：新用户无需帮助即可完成首次分析
3. **品牌感知**：用户认为产品具有"科技感"和"艺术感"

### 验收标准
- [ ] 所有页面完成重新设计并通过设计评审
- [ ] 核心用户旅程测试通过率 ≥ 95%
- [ ] 无重大 UI bug（控件重叠、文字截断等）
- [ ] 性能指标达标（响应时间、帧率）
- [ ] 通过无障碍测试

## Constraints & Assumptions

### 技术约束
- 必须使用 Flutter 框架
- 参考 Flutter-Neumorphic UI 库风格
- 保持与现有 MediaPipe 集成的兼容性
- 本地存储使用 SQLite

### 时间约束
- 设计阶段：2 周
- 开发阶段：4-6 周
- 测试与优化：2 周
- 总计：8-10 周

### 资源约束
- 开发团队：1-2 名 Flutter 开发者
- 设计资源：需要 UI/UX 设计师支持
- 测试资源：需要真实用户参与 Beta 测试

### 假设
- 用户设备性能足以支持流畅动画
- 用户已授予相机和相册访问权限
- 网络连接稳定（未来云同步功能）
- 用户熟悉基本的移动应用操作

## Out of Scope

本次重新设计 **不包括** 以下内容：

1. **后端功能**：云同步、用户账号系统（V2 阶段）
2. **高级分析**：骨骼叠加可视化、关键帧标注（V2 阶段）
3. **社交功能**：分享到社交媒体、社区互动
4. **教练模式**：训练计划、周期报告（V4 阶段）
5. **IMU 集成**：蓝牙传感器支持（V3 阶段）
6. **多语言支持**：仅支持中文（初期）
7. **Web 版本**：仅移动端（iOS/Android）

## Dependencies

### 外部依赖
1. **Flutter-Neumorphic 库**：需要评估和集成
2. **系统相册 API**：依赖 iOS Photos 和 Android MediaStore
3. **设计资源**：需要 UI/UX 设计师提供设计稿和资源
4. **测试设备**：需要多种型号的 iOS 和 Android 设备

### 内部依赖
1. **MediaPipe 集成**：UI 需要与现有分析引擎无缝对接
2. **数据模型**：历史记录和结果展示依赖现有数据结构
3. **性能优化**：UI 流畅度依赖底层性能优化

### 团队依赖
1. **设计团队**：提供完整的设计系统和视觉规范
2. **开发团队**：实现设计并保证性能
3. **测试团队**：进行全面的功能和体验测试
4. **产品团队**：收集用户反馈并迭代优化

## Risks & Mitigation

### 风险 1：设计与开发脱节
**影响**：设计稿无法实现或性能不达标
**缓解措施**：设计阶段引入技术评审，确保可实现性

### 风险 2：性能问题
**影响**：动画卡顿，影响用户体验
**缓解措施**：早期进行性能测试，优化关键路径

### 风险 3：用户不适应新界面
**影响**：老用户流失
**缓解措施**：提供引导教程，逐步灰度发布

### 风险 4：开发周期延长
**影响**：延迟上线，错过市场窗口
**缓解措施**：采用 MVP 方式，优先核心功能，次要功能后续迭代
