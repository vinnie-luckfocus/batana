# PRD: mvp-core-analysis

## Executive Summary

mvp-core-analysis 是 batana 项目的第一阶段交付，目标是验证"用户拍摄挥棒视频即可获得量化评分与改进建议"这一核心价值假设。通过 Flutter 跨平台框架 + Google ML Kit Pose Detection 本地推理 + 规则评分引擎，实现从视频录制到分析结果展示的完整闭环，为后续迭代奠定技术与产品基础。

MVP 阶段聚焦最小可用功能集：单次挥棒录制、2D 姿态识别、3 个核心技术指标（速度/角度/协调性）、综合评分（0-100）、2 条以上可执行建议、最近 10 次本地历史记录。不追求完美精度，优先保证流程可用性与反馈可解释性。

## 阶段退出门槛（Go/No-Go）

MVP 阶段必须满足以下硬性 KPI 方可进入 V2：
1. 端到端分析成功率 ≥ 95%（标准拍摄条件下）
2. 分析完成率 ≥ 85%（录制后成功生成结果）
3. D7 留存率 ≥ 25%
4. 崩溃率 < 1%
5. 核心算法单元测试覆盖率 ≥ 80%

## Problem Statement

### 我们正在解决什么问题
棒球训练者（入门爱好者、青少年球员）缺乏低成本、可持续的挥棒动作反馈渠道：
1. 自我感知不准确，无法量化进步
2. 教练指导成本高、频次低
3. 普通视频回放难以定位具体问题

MVP 要验证：用户是否愿意用手机自主拍摄并接受 AI 分析结果作为训练参考。

### 为什么现在必须解决
1. 移动端 AI 推理能力已成熟（Google ML Kit 等官方方案可用）
2. 用户对"数据化训练"接受度提升
3. 需要快速验证市场需求，避免过度投入

## User Stories

### 目标用户
**Persona：棒球入门爱好者**
- 年龄：15-35 岁
- 场景：在家、球场边自主练习
- 需求：知道"这次挥棒好不好"、"哪里需要改进"
- 痛点：不知道自己错在哪，训练缺乏方向

### 核心用户故事

**US-1：快速获得分析结果**
- 作为入门用户，我希望拍摄一段挥棒视频后在 15 秒内得到评分和建议，以便快速了解表现。
- 验收标准：
  - 录制时长 10-15 秒
  - 分析耗时 ≤ 15 秒
  - 输出综合评分（0-100）
  - 输出至少 2 条可执行建议（如"重心后移不足"）

**US-2：查看历史记录**
- 作为入门用户，我希望查看最近的训练记录，以便对比今天和昨天的表现。
- 验收标准：
  - 本地保存最近 10 次分析记录
  - 显示日期、评分、关键指标
  - 可点击查看详情

**US-3：理解分析结果**
- 作为入门用户，我希望看到通俗易懂的问题描述和改进建议，而不是专业术语。
- 验收标准：
  - 建议使用"通俗解释 + 可执行动作"格式
  - 避免纯技术术语（如"髋肩分离角度不足"需配通俗说明）

## Requirements

### Functional Requirements

#### FR-1 视频录制与引导
- 支持移动端摄像头录制 10-15 秒视频
- 提供录制引导：
  - 站位示意图（正侧面 45°）
  - 拍摄距离建议（2-3 米）
  - 光照检测提示
- 支持录制后预览与重拍

#### FR-2 姿态识别
- 集成 Google ML Kit Pose Detection 进行 2D 关键点检测
- 识别关键身体部位：头、肩、肘、腕、髋、膝、踝（33 个关键点）
- 输出每帧关键点坐标序列
- 处理遮挡与识别失败情况（提示用户重拍）
- 技术实现：`google_mlkit_pose_detection` 插件，支持 33 个身体关键点

#### FR-3 挥棒阶段分割
- 自动识别挥棒动作阶段：
  - 准备期（站位到启动）
  - 加速期（启动到击球点）
  - 击球期（击球瞬间）
  - 收尾期（击球后减速）
- 基于腕部速度峰值检测关键时间点
- 阶段分割失败时的降级策略：
  - 若无法识别完整 4 阶段，尝试识别"启动-击球-收尾"3 阶段
  - 若仍失败，提示用户"未检测到完整挥棒动作，请重新录制"并给出具体原因（如"未检测到明显挥棒动作"或"视频中存在多次挥棒，请仅录制一次"）

#### FR-3.1 采集质量前置门控
- 在分析前自动评估视频质量：
  - **光照检测**：画面亮度低于阈值时提示"光线不足，请在明亮环境下录制"
  - **人体检测**：未检测到完整人体时提示"请确保全身在画面内"
  - **稳定性检测**：画面抖动过大时提示"请固定手机或使用三脚架"
  - **角度检测**：人体朝向不符合要求时提示"请从侧面 45° 角拍摄"
- 质量评分 < 60 分时阻止分析并给出具体改进提示
- 质量评分 60-80 分时允许分析但标注"拍摄条件一般，结果仅供参考"

#### FR-4 核心指标计算
- **挥棒速度**：腕部轨迹速度峰值（像素/帧 × 标定系数）
  - 标定流程：用户首次使用时输入身高，系统基于肩宽像素值与人体比例估算像素-实际距离映射
  - 输出单位：相对速度评级（慢/中/快），不输出绝对速度值（单目视觉精度不足以支撑）
- **挥棒角度**：挥棒平面与水平面夹角
- **动作协调性**：髋肩启动时序差、重心转移流畅度
- 所有指标均为视觉估算值，结果页标注"基于视觉分析的参考值"

#### FR-5 评分与反馈
- 综合评分（0-100）：基于 3 个核心指标加权计算
- 问题诊断：识别常见错误模式（如"过早开肩"、"重心后移不足"）
- 改进建议：每次输出 2-3 条可执行建议

#### FR-6 本地存储
- 使用 SQLite 存储分析记录
- 保存最近 10 次记录（超出自动删除最旧）
- 存储内容：日期、视频缩略图、评分、指标、建议

#### FR-7 结果展示
- 分析结果页：
  - 综合评分（大号数字 + 进度环）
  - 3 个核心指标（图标 + 数值 + 简短说明）
  - 改进建议列表
- 历史记录页：
  - 时间倒序列表
  - 每条显示日期、评分、缩略图
  - 点击查看详情

#### FR-8 数据埋点与指标口径
- 核心事件埋点：
  - `app_open`：应用启动
  - `user_first_open`：用户首次启动（标记新增用户）
  - `video_record_start`：开始录制
  - `video_record_complete`：录制完成
  - `analysis_start`：开始分析
  - `analysis_complete`：分析成功
  - `analysis_failed`：分析失败（含失败原因）
  - `result_view`：查看分析结果
  - `result_helpful_rate`：用户对结果有帮助程度评分（1-5 分，结果页弹窗）
  - `history_view`：查看历史记录
  - `app_crash`：应用崩溃（自动上报）
- KPI 计算口径：
  - **分析完成率** = `analysis_complete` / `video_record_complete`
  - **D7 留存率** = 第 7 天触发 `app_open` 的用户数 / 第 1 天触发 `user_first_open` 的用户数
  - **用户"结果有帮助"主观评分** = `result_helpful_rate` 评分平均值（1-5 分）
  - **每周人均分析次数** = 周内 `analysis_complete` 总数 / 周活跃用户数（周内至少 1 次 `app_open`）
  - **端到端分析成功率** = `analysis_complete` / `analysis_start`（标准拍摄条件下采样）
  - **崩溃率** = `app_crash` 会话数 / 总会话数（会话定义：从 `app_open` 到应用退出）
  - **单元测试覆盖率**：通过 CI 工具（如 Codecov）自动统计核心算法模块覆盖率，数据来源为 CI 构建日志，验收标准为 CI 报告显示覆盖率 ≥ 80%
- 埋点实施责任：客户端 SDK 集成（如 Firebase Analytics / 自建）+ CI 工具集成

### Non-Functional Requirements

#### NFR-1 性能
- 分析结果生成时间：P50 ≤ 10 秒，P95 ≤ 15 秒（基准机型：Snapdragon 7 系列 / A14 级别）
- 应用启动时间 ≤ 3 秒（冷启动）
- 录制流畅度 ≥ 30 FPS
- 热降频保护：连续分析 3 次后提示"设备需要冷却"，避免因 CPU 降频导致分析超时

#### NFR-2 可用性
- 核心流程完成率 ≥ 85%（定义：从点击"开始录制"到看到分析结果的用户比例）
- 新用户首次完成分析操作步骤 ≤ 5 步（录制→确认→分析→查看结果）
- 分析失败时提供可理解的错误反馈与重试引导

#### NFR-3 准确性
- 关键点识别成功率 ≥ 85%（标准拍摄条件：室外自然光/室内充足照明，侧面 45°，2-3 米距离，全身入画）
- 同一动作重复测试评分波动 ≤ 10 分（标准差）

#### NFR-4 兼容性
- 支持 iOS 15.5+ / Android 9+（覆盖 90%+ 活跃设备）
- 适配主流屏幕尺寸（5-7 英寸）
- 基准测试机型：iPhone 12 / Pixel 6 / 三星 Galaxy A53
- ML Kit 集成：使用 `google_mlkit_pose_detection` 插件（v0.12.0）

#### NFR-5 隐私
- 视频仅本地存储，不上传云端
- 用户可随时删除单条或全部历史记录
- 不采集可识别身份信息（PII），仅采集最小必要训练参数（如身高用于速度标定）且仅本地存储
- 未成年人场景说明：MVP 阶段纯本地应用，不涉及账号注册与数据上传，不触发 COPPA 合规要求；但 Persona 覆盖 15 岁用户，需在应用商店描述中标注适用年龄（13+），并在隐私政策中声明"不主动采集未成年人个人身份信息"

## Constraints & Assumptions

### 技术约束
1. 单目视觉深度信息缺失，速度为估算值
2. 光照、背景、遮挡会影响识别质量
3. 需要用户按引导拍摄（正侧面角度）

### 资源约束
1. MVP 阶段 2-3 个月交付
2. 优先保证流程可用，不追求极限精度

### 核心假设
1. 用户愿意接受"可解释但非完美"的早期结果
2. 规则评分引擎足以支撑 MVP 阶段反馈
3. 本地存储满足 MVP 需求，暂不需要云端

#### 2. 视频采集 (`lib/capture/`)
- `camera_preview.dart` - 摄像头预览组件
- `video_recorder.dart` - 视频录制管理
- `quality_gate.dart` - 采集质量前置门控

#### 3. 评分引擎 (`lib/scoring/`)
- `scoring_engine.dart` - 综合评分计算
- `problem_detector.dart` - 问题模式识别
- `suggestion_generator.dart` - 改进建议生成

#### 4. 数据存储 (`lib/storage/`)
- `database.dart` - SQLite 数据库管理
- `database_manager.dart` - 数据库操作封装

### 图像处理流程
1. 摄像头采集 `CameraImage` (YUV420/NV21 格式)
2. 转换为 ML Kit `InputImage` 格式
3. ML Kit Pose Detector 处理 → 33 个关键点
4. 关键点序列 → 阶段分割 → 指标计算 → 评分

## Risks & Mitigation

### 主要风险
1. **ML Kit 识别精度不足** → 先用固定角度拍摄降低难度；使用 Accurate 模型提升精度（设备性能允许时）
2. **评分结果不可信** → 建立"可解释性"优先于"精度"；所有指标标注"参考值"
3. **用户不理解如何拍摄** → 强化录制引导与示例视频；前置质量门控拦截低质量视频
4. **机型差异导致性能波动** → 建立基准测试机型矩阵（高/中/低端各 2 款），P95 指标覆盖中端机型
5. **权限获取失败** → 摄像头/存储权限被拒时提供清晰引导，说明权限用途并支持跳转系统设置
6. **热降频导致分析超时** → 连续分析场景增加冷却提示；通过监控分析耗时与帧率变化检测性能下降（不直接读取 CPU 温度，因平台 API 限制）

### 缓解策略
1. 提供清晰的拍摄引导与质量检测前置门控
2. 建立"通俗解释 + 可执行建议"反馈结构
3. 灰度测试收集真实反馈快速迭代
4. 建立多机型 CI 测试流水线

## 失败模式与降级策略

| 失败场景 | 降级策略 | 用户提示 |
|---------|---------|---------|
| 关键点识别失败 | 跳过分析，引导重拍 | "未能识别动作，请确保全身在画面内并重新录制" |
| 阶段分割失败 | 尝试 3 阶段简化分割 | "动作识别不完整，评分仅供参考" |
| 分析超时（>20秒） | 终止分析，释放资源 | "分析超时，请关闭其他应用后重试" |
| 存储空间不足 | 阻止录制，提示清理 | "存储空间不足，请清理后重试" |
| ML Kit 加载失败 | 提示重启应用 | "分析引擎加载失败，请重启应用" |
